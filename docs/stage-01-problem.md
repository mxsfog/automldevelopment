# Стадия 01: Постановка задачи (Problem Definition)

**Проект:** Universal AutoResearch Framework (UAF)
**Дата:** 2026-03-19
**Статус:** STAGE COMPLETE
**Автор:** ML-инженер (запуск вручную, локальная машина)

---

## 1. Inverted Pyramid: Бизнес-контекст -> Задача -> Ограничения

### 1.1 Бизнес-контекст (верхний уровень)

ML-инженер регулярно решает новые исследовательские задачи: подбирает архитектуру,
выбирает признаки, настраивает гиперпараметры, анализирует ошибки модели.
Каждый цикл экспериментирования включает ручное написание кода, запуск, анализ
результатов, формулировку следующей гипотезы. Это медленно и требует постоянного
присутствия человека.

**Бизнес-боль:** цикл "гипотеза -> эксперимент -> вывод -> следующая гипотеза"
занимает часы и дни из-за ручного труда на каждом шаге. Человек вынужден
переключаться между написанием кода, ожиданием результатов и интерпретацией,
что разрушает глубокое погружение и замедляет итерации.

**Ценность автоматизации:** если система может автономно генерировать план
исследования, писать код экспериментов, запускать их, анализировать результаты
и формулировать следующие шаги — ML-инженер получает возможность работать над
несколькими задачами параллельно или делегировать рутинные исследования системе
целиком, фокусируясь только на финальном решении.

### 1.2 Задача (средний уровень)

Построить фреймворк, который автоматизирует цикл ML-исследований:
от постановки задачи через LLM-планирование, автоматическое написание кода,
запуск экспериментов с трекингом (MLflow), анализ результатов до
формулировки следующих шагов или финального отчёта.

**Тип задачи по классификации ML-систем:**
- Это не ML-модель. Это ML-система управления экспериментами с LLM-агентами
  внутри.
- Ключевой компонент: LLM как "мозг" планировщика (агент-проектировщик),
  инструменты как "руки" (codegen, executor, MLflow, DVC).
- Паттерн: ReAct/Plan-and-Execute агент с циклом обратной связи.

**Конкретные сценарии использования:**

1. **Сценарий A — Research Cycle:** Пользователь описывает ML-задачу и датасет.
   Система генерирует `program.md` (план исследования), пользователь правит план,
   система выполняет эксперименты по плану, возвращает отчёт.

2. **Сценарий B — Hypothesis Testing:** Пользователь задаёт конкретную гипотезу
   (например, "проверить, помогает ли добавление признака X"). Система пишет код
   эксперимента, запускает, сравнивает с baseline в MLflow, возвращает вывод.

3. **Сценарий C — Architecture Search:** Пользователь задаёт задачу и метрику.
   Система итеративно пробует разные архитектуры/подходы, логирует все в MLflow,
   возвращает сравнительную таблицу и рекомендацию.

### 1.3 Ограничения (нижний уровень)

**Технические ограничения:**
- Запуск только на локальной машине пользователя (нет облака, нет кластера)
- Один GPU (или CPU), нет параллельного запуска на нескольких машинах
- MLflow обязателен для трекинга всех экспериментов
- DVC обязателен для версионирования данных и артефактов
- LLM-агент — внешний API (OpenAI/Anthropic/etc.) или локальная модель
- Python 3.10+, PyTorch для DL-задач, CatBoost/XGBoost для таблиц

**Операционные ограничения:**
- Пользователь — один человек, ML-инженер
- Нет CI/CD, нет автодеплоя, нет продакшена
- Система запускается вручную, результат возвращается пользователю
- Нет требований к latency (batch-режим исследований, не realtime)

**Бюджетные ограничения:**
- Поддерживаются два режима: фиксированный бюджет (max N API-вызовов / M рублей)
  и динамический бюджет (система сама определяет сколько итераций нужно)
- В фиксированном режиме: жёсткий стоп при достижении лимита, отчёт по
  достигнутому
- В динамическом режиме: система останавливается при достижении сходимости
  или явной бесполезности дальнейших итераций

## 2. Определение успеха

### 2.1 Что такое "успешный запуск системы"

Пользователь запустил UAF с описанием задачи -> система вернула:
1. Заполненный `program.md` с планом исследования (LLM-черновик, готовый к правке)
2. Результаты выполненных экспериментов в MLflow (доступны через UI)
3. Артефакты зафиксированы в DVC
4. Финальный отчёт с выводами и рекомендацией по следующим шагам

### 2.2 Критерии качества исследования

- Каждый эксперимент воспроизводим: зафиксированы seed, версия данных, код,
  гиперпараметры
- Все запуски сравнимы между собой через MLflow
- Сгенерированный код запускается без ошибок с вероятностью >= 80% с первой
  попытки (или система сама исправляет ошибки)
- Итоговый отчёт содержит явный вывод: "рекомендуется X, потому что Y"

### 2.3 Критерии качества плана (program.md)

- Структурирован: каждый шаг имеет цель, метрику успеха, критерий остановки
- Покрывает baseline -> итеративные улучшения -> финальное сравнение
- Содержит явные зависимости между шагами
- Человеку достаточно 5-15 минут чтобы проверить и одобрить план

## 3. Antigoals (что система НЕ должна делать)

**Antigoal 1. Система НЕ является AutoML-инструментом для production-деплоя.**
UAF исследует и экспериментирует, но не готовит модели к продакшену, не
оптимизирует inference, не строит serving-пайплайны. Как только исследование
завершено — работа UAF закончена. Всё что после — вне scope.

**Antigoal 2. Система НЕ должна запускать эксперименты без одобрения человека
на этапе плана.**
Автономный запуск без проверки `program.md` человеком недопустим в стандартном
режиме. Причина: LLM может генерировать план с неправильной постановкой задачи,
неподходящими метриками или бессмысленными гипотезами. Дорогостоящие эксперименты
(по времени и API-бюджету) не должны стартовать без человеческого одобрения плана.
(Исключение: явный "fully_autonomous" режим, включаемый пользователем намеренно.)

**Antigoal 3. Система НЕ должна скрывать неудачные эксперименты или
"подбирать" результаты.**
Все запуски, включая упавшие и показавшие плохие метрики, должны быть
залогированы в MLflow с полными метаданными. Система не должна фильтровать
"неудобные" результаты из отчёта или представлять только лучший результат
без контекста остальных попыток.

**Antigoal 4. Система НЕ должна самостоятельно изменять пользовательские данные
или исходный код проекта.**
UAF работает в своей директории экспериментов. Она может читать данные
пользователя (только чтение), но не может модифицировать оригинальные датасеты,
исходный код проекта или любые файлы за пределами своей рабочей директории без
явного разрешения.

**Antigoal 5. Система НЕ должна оптимизировать под конкретный датасет в ущерб
обобщаемости кода.**
Сгенерированный код экспериментов должен быть читаемым и переиспользуемым, а не
"магическим" набором хаков под конкретный датасет. Цель — исследование,
которое человек может понять и воспроизвести, а не чёрный ящик с хорошей метрикой.

**Antigoal 6. Система НЕ должна превышать заданный бюджет в фиксированном режиме.**
В режиме фиксированного бюджета жёсткий лимит — не рекомендация, а hard constraint.
Система должна останавливаться и возвращать лучший найденный результат до
исчерпания бюджета, а не превышать лимит "ради хорошего результата".

| Роль | Имя/описание | Интерес |
|------|-------------|---------|
| Владелец и пользователь | ML-инженер | Ускорение исследований, меньше ручного труда |

## 5. Текущий процесс (As-Is) vs. Целевой процесс (To-Be)

### As-Is (без UAF):
```
Человек формулирует гипотезу
  -> Человек пишет код эксперимента (30-120 мин)
  -> Человек запускает эксперимент (ожидание)
  -> Человек анализирует результаты (15-60 мин)
  -> Человек формулирует следующую гипотезу
  -> Повторить
Итого на цикл: 1-4 часа активного времени + время ожидания
```

### To-Be (с UAF):
```
Человек описывает задачу (5-15 мин)
  -> UAF генерирует program.md (автомат, 2-5 мин)
  -> Человек проверяет и одобряет план (5-15 мин)
  -> UAF выполняет эксперименты по плану (автомат, ожидание)
  -> UAF формирует отчёт с выводами (автомат)
  -> Человек читает отчёт и принимает решение (10-20 мин)
Итого активного времени человека: 20-50 мин на весь цикл исследования
```

**Ожидаемый выигрыш:** 5-10x сокращение активного времени ML-инженера на
рутинные исследовательские циклы.

## 6. Ключевые риски стадии 01

| # | Риск | Вероятность | Влияние | Митигация |
|---|------|-------------|---------|-----------|
| R1 | LLM генерирует некорректные планы, человек не замечает | Средняя | Высокое | Обязательное одобрение плана, структурированный формат program.md |
| R2 | Сгенерированный код не запускается, система зависает | Высокая | Среднее | Retry-логика, self-correction агент, таймауты |
| R3 | Превышение API-бюджета в динамическом режиме | Средняя | Среднее | Soft warning на 80% бюджета, hard stop на 100% |
| R4 | Воспроизводимость нарушена из-за внешних зависимостей | Средняя | Высокое | DVC для данных, pinned зависимости, seed фиксация |
| R5 | Scope creep — система начинает делать "слишком много" | Низкая | Высокое | Строгие antigoals, явный scope в каждой стадии |

## 7. Связь с методологией

**Inverted Pyramid применён:**
- Уровень 1 (бизнес): ускорение исследовательских циклов ML-инженера
- Уровень 2 (задача): автономный агент планирования и выполнения ML-экспериментов
- Уровень 3 (ограничения): локально, один пользователь, MLflow+DVC обязательны,
  два режима бюджета

**Antigoals определены:** 6 штук, покрывают scope, безопасность данных,
прозрачность результатов, бюджетный контроль.

**Следующая стадия:** 02-research (обзор существующих решений: AutoML-фреймворки,
LLM-агенты для кода, research automation инструменты).
